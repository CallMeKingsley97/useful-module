# 需求文档：路由命中卡小组件

## 1. 背景与目标
Egern 的代理用户并不缺“当前节点名称”，真正高频的问题是：**这段时间流量怎么走、规则有没有按预期工作、拒绝是否异常升高**。路由命中卡的目标不是做一个开发者面板，而是在主屏一眼回答这三个问题。

**核心目标：**
- 在尽量少的信息里展示近一段时间的直连、代理、拒绝占比
- 用最近高频命中的规则组帮助用户判断规则是否跑偏
- 用明显但克制的状态提示告知是否存在异常拒绝或采样过旧
- 设计风格贴合 iOS 主屏环境，像 Apple 自带信息卡，而不是网络调试面板

## 2. 用户画像与使用场景
### 2.1 用户画像
- 规则型代理用户：关心分流是否按预期工作
- 轻度折腾用户：不看复杂日志，但想快速知道“现在大概都走哪条路”
- 主屏审美用户：希望小组件能长期驻留，而不是只在排障时临时放上去

### 2.2 使用场景
- 早上查看主屏时，快速确认今天大部分请求是否仍走代理
- 切换配置或更新规则后，观察最近 30 分钟命中是否符合预期
- 某个服务异常时，先看拒绝占比和最近命中规则组，再决定是否深入排障

## 3. Egern 能力边界与数据来源
### 3.1 能力边界
- `generic` 小组件脚本可使用 `ctx.env`、`ctx.storage`、`ctx.http`、`ctx.widgetFamily`
- Egern 当前公开 JavaScript API 文档未明确提供“全局实际分流结果统计”读取接口，因此小组件**不能直接**读取全局直连/代理/拒绝计数
- v1 必须通过 **companion script -> `ctx.storage` -> widget** 的链路实现

### 3.2 数据模式
本组件定义两种数据模式，避免做成伪需求：

1. **观察模式（推荐）**
   - 由 companion `request/response/network` 脚本在可观测流量路径上记录路由结果
   - 适合用户愿意在自己的规则链路中挂统计脚本
   - 小组件展示真实采样结果

2. **采样模式（降级）**
   - 由 `schedule` 脚本按固定间隔对预设站点进行主动探测
   - 通过不同 `policy` 与请求结果估算“直连/代理/拒绝”结构
   - 适合拿不到全局流量事件时的近似方案
   - 文案必须显示“采样视角”，避免误导成全量统计

### 3.3 推荐 companion script 组合
- `request` 脚本：在可控流量链路内记录命中规则组、目标域名、推断路由类别
- `schedule` 脚本：负责窗口压缩、Top 规则组聚合、异常阈值判断
- `generic` 脚本：只做读取、格式化和多尺寸渲染

## 4. 产品定义
### 4.1 一句话卖点
把“规则是否正常工作”翻译成一张一眼看懂的玻璃卡片。

### 4.2 核心输出
- 近 30 分钟直连/代理/拒绝占比
- 最近命中最多的 3 个规则组
- 当前采样窗口与最近刷新状态
- 异常提示：拒绝偏高、样本过少、数据过旧

### 4.3 成功标准
- 用户在 1 秒内看懂“现在大多数请求走哪条路”
- 用户在 3 秒内看懂“规则是否疑似异常”
- Small 尺寸不出现拥挤感或安卓式堆料感

## 5. 数据契约
### 5.1 推荐 storage key
- `routing-hit-summary`

### 5.2 推荐 storage 结构
```json
{
  "mode": "observed",
  "windowMinutes": 30,
  "sampleStartAt": "2026-03-18T09:30:00+08:00",
  "sampleEndAt": "2026-03-18T10:00:00+08:00",
  "updatedAt": "2026-03-18T10:00:00+08:00",
  "totals": {
    "directCount": 124,
    "proxyCount": 286,
    "rejectCount": 18
  },
  "recentRules": [
    { "name": "OpenAI", "count": 78, "route": "proxy" },
    { "name": "YouTube", "count": 64, "route": "proxy" },
    { "name": "广告拦截", "count": 18, "route": "reject" }
  ],
  "trend": [
    { "slot": "09:35", "direct": 8, "proxy": 19, "reject": 1 },
    { "slot": "09:40", "direct": 12, "proxy": 27, "reject": 2 }
  ],
  "flags": {
    "stale": false,
    "lowSample": false,
    "rejectSpike": false
  }
}
```

### 5.3 字段规则
- `mode`：`observed` 或 `sampled`
- `windowMinutes`：默认 `30`
- `trend`：用于 Large 的最近 6 个 5 分钟片段；不足可为空
- `flags.stale`：超过 `2 * REFRESH_MINUTES` 未更新则为 `true`
- `flags.lowSample`：总样本数 `< MIN_SAMPLE_COUNT` 时为 `true`
- `flags.rejectSpike`：`rejectCount / totalCount >= REJECT_ALERT_THRESHOLD`

## 6. 功能需求
### 6.1 核心功能
- 读取 storage 中最近窗口的汇总结果
- 计算三类路由占比并生成主视觉比例
- 展示 Top 3 规则组，优先显示名称与样本数
- 根据异常状态输出中文状态文案

### 6.2 增强功能
- 当 `mode=sampled` 时，在底部状态区明确显示“采样视角”
- Large 尺寸展示简化趋势条或趋势点阵
- 支持点击小组件跳转到 Egern 指定页面或外链帮助文档

### 6.3 不纳入 v1
- 全局精确分流日志浏览
- 规则级历史回溯明细
- 用户在小组件内直接切换策略或修改规则

## 7. 信息架构与多尺寸布局
### 7.1 统一信息层级
- 主焦点：路由占比
- 辅助指标：Top 规则组、采样窗口
- 状态提示：拒绝异常 / 采样过旧 / 数据模式

### 7.2 `systemSmall`
- 顶部：标题或短标签，如“路由命中”
- 中部：单个主焦点占比图形，优先展示代理占比与一句摘要
- 底部：`直连 29% / 拒绝 4%` 或 `采样视角`

建议文案：
- `大多数请求正走代理`
- `拒绝偏高，建议检查规则`

### 7.3 `systemMedium`
- 左侧：玻璃圆环或分段条，突出三类占比
- 右侧：Top 3 规则组标签流，每项包含名称和命中次数
- 底部：采样窗口、刷新状态、异常提示

### 7.4 `systemLarge`
- 顶部：标题、窗口时间、状态徽标
- 中部：左侧占比结构，右侧最近高频规则组
- 底部：最近 6 个采样片段趋势、异常解释文案

### 7.5 `accessoryCircular`
- 仅在 `ALLOW_CIRCULAR=true` 时提供
- 中心显示代理占比或拒绝占比
- 圆环颜色根据主状态切换

### 7.6 `accessoryRectangular`
- 第一行：`代理 67% · 直连 29%`
- 第二行：`最近：OpenAI / YouTube`

### 7.7 `accessoryInline`
- 单行：`路由命中：代理 67%，拒绝 4%`

## 8. 视觉规范
### 8.1 视觉方向
- 风格：Apple 风格玻璃卡片
- 气质：清透、克制、信息优先
- 避免：高饱和霓虹、厚重描边、仪表盘式大面积图形

### 8.2 主色板
- 代理：`#5B8CFF`
- 直连：`#5CC48D`
- 拒绝：`#FF7D6E`
- 中性文字：`#F5F7FA` / `#C7D0DD`
- 玻璃底：`rgba(18, 28, 45, 0.72)` 语义等价色

### 8.3 卡片层次
- 背景：低饱和蓝灰渐变
- 主内容层：半透明信息层，允许轻微高光
- 状态层：小面积色块或彩色文字，不用满屏状态色

### 8.4 字号层级
- 标题：`title3` 或 `headline`
- 主指标：`title2` 到 `largeTitle` 之间按尺寸缩放
- 规则组标签：`caption` 到 `callout`
- 状态文案：`caption2`

### 8.5 圆角、边框、阴影
- 主卡圆角偏大，贴近 iOS 卡片感
- 边框仅用 1px 级低透明高光边
- 阴影只做轻微景深，不做厚重投影

### 8.6 留白规则
- Small：内边距 14-16
- Medium：内边距 16-18
- Large：内边距 18-22
- accessory：优先压缩装饰层，不压主信息

## 9. 环境变量（Env）
- `TITLE`：默认 `路由命中`
- `REFRESH_MINUTES`：默认 `15`
- `OPEN_URL`：点击跳转地址
- `ACCENT_MODE`：`auto` / `cool` / `neutral`
- `STORAGE_KEY`：默认 `routing-hit-summary`
- `DATA_MODE`：`observed` / `sampled` / `auto`
- `SAMPLE_WINDOW_MINUTES`：默认 `30`
- `MIN_SAMPLE_COUNT`：默认 `40`
- `REJECT_ALERT_THRESHOLD`：默认 `0.15`
- `MAX_RULES`：默认 `3`
- `ALLOW_CIRCULAR`：默认 `false`

## 10. 状态与异常处理
### 10.1 正常状态
- 展示占比、Top 规则组、窗口信息

### 10.2 无样本
- 主文案：`还没有可用样本`
- 次文案：`等待 companion script 写入统计`
- 视觉上保留玻璃底，不显示空图表

### 10.3 样本过旧
- 主文案不变
- 状态区增加：`数据已过期`
- 颜色降为中性灰蓝，避免误导为实时结果

### 10.4 规则组名称过长
- Small/Medium 截断到 8 个中文字符或 16 个英文字符
- Large 可放宽到 12 个中文字符
- 完整名只在 companion 页面或跳转页查看，不在 widget 中硬塞

### 10.5 拒绝占比异常升高
- 若 `rejectSpike=true`，状态标签改为暖红色
- 文案：`拒绝偏高，优先检查拦截规则`
- Small 不额外增加行数，只替换原状态行

## 11. 验收标准
### 11.1 尺寸验收
- `systemSmall / systemMedium / systemLarge / accessoryRectangular / accessoryInline` 必须成立
- `accessoryCircular` 仅在开启后提供，且数字不拥挤

### 11.2 数据验收
- 正常数据：占比、规则组、状态正确显示
- 空数据：显示等待态
- 脏数据：非法计数按 0 处理并标记低样本
- 过期数据：显示过期提示并弱化状态色

### 11.3 审美验收
- 首屏 1 秒内能看懂主比例
- 截图放在 iPhone 主屏时不显得“第三方监控工具味过重”
- 高亮色只落在关键状态，不抢主体信息

### 11.4 实现验收
- 文档可直接映射到 `generic` + companion 脚本 + `storage` 契约
- 没有要求 generic 脚本自行统计全量流量

## 12. 开发拆解建议
1. 实现 companion 统计脚本，明确观察模式与采样模式写入结构
2. 实现窗口聚合与异常 flag 计算
3. 实现 widget view model：占比、规则标签、状态文案
4. 分别实现 Small / Medium / Large / accessory 布局
5. 验证长规则名、低样本、拒绝异常、过期数据四类边界
