# 需求文档：出口身份卡小组件

## 1. 背景与目标
代理用户经常会问一个非常直接的问题：**我现在到底从哪里出去？** 出口身份卡的目标是把地域、IP、ASN、DNS 与当前网络环境整理成一张清爽的地点卡，而不是冷冰冰的技术字段堆叠。

**核心目标：**
- 清楚展示当前出口地区与 IP 身份
- 给出 ASN/运营商和 DNS 摘要，帮助判断是否符合预期
- 当用户设置“期望地区”时，明确提示是否匹配
- 视觉上更像 Apple Weather 的地点卡，而不是 IP 查询网站截图

## 2. 用户画像与使用场景
### 2.1 用户画像
- 流媒体用户：关注出口地区是否正确
- AI 服务用户：需要快速确认出口身份是否稳定
- 网络折腾用户：会关心 ASN、DNS、Wi‑Fi/蜂窝环境

### 2.2 使用场景
- 切换策略后看一眼主屏，确认是否已经出到预期地区
- 某服务不可用时，先看出口地区和 DNS 是否异常
- 漫游、热点、蜂窝切换后，确认当前网络环境与出口是否一致

## 3. Egern 能力边界与数据来源
### 3.1 可直接获取的数据
- `ctx.device.dnsServers`
- `ctx.device.wifi.ssid`
- `ctx.device.cellular.carrier`
- `ctx.device.ipv4.address` / `ctx.device.ipv6.address`

### 3.2 需要网络查询的数据
- 公网出口 IP
- 城市 / 地区 / 国家
- ASN / 运营商

### 3.3 查询方案
- `schedule` 脚本使用 `ctx.http.get(IP_INFO_API, { policy })` 查询出口身份
- 将最近一次成功结果写入 `ctx.storage`
- `generic` 小组件优先读取缓存，避免每次渲染都发起网络请求

### 3.4 推荐查询源
- 默认优先使用返回轻量 JSON 的 IP 信息服务
- 接口需至少返回：IP、城市、地区、国家、ASN 或 ISP
- 若主接口失败，可配置备用接口

## 4. 产品定义
### 4.1 一句话卖点
让“现在从哪里出去”变成一张带地理感的苹果式玻璃卡片。

### 4.2 核心输出
- 出口地区与城市
- 出口 IP
- ASN / 运营商
- DNS 摘要
- 当前网络环境（Wi‑Fi / 蜂窝）
- 与期望地区是否一致

## 5. 数据契约
### 5.1 推荐 storage key
- `egress-identity-summary`

### 5.2 推荐 storage 结构
```json
{
  "policyName": "Proxy",
  "updatedAt": "2026-03-18T10:00:00+08:00",
  "ttlMinutes": 30,
  "identity": {
    "ip": "203.0.113.18",
    "city": "Tokyo",
    "region": "Tokyo",
    "country": "Japan",
    "countryCode": "JP",
    "asn": "AS2516",
    "org": "KDDI CORPORATION"
  },
  "device": {
    "dnsServers": ["1.1.1.1", "8.8.8.8"],
    "wifiSsid": "OfficeWiFi",
    "carrier": null,
    "localIpv4": "192.168.50.20"
  },
  "expectation": {
    "expectedRegion": "Japan",
    "matched": true
  },
  "flags": {
    "stale": false,
    "partialData": false,
    "queryFailed": false
  }
}
```

### 5.3 字段规则
- `ttlMinutes`：默认 `30`
- `matched`：按 `EXPECTED_REGION` 与 `country/region/city` 模糊比较
- `partialData`：缺失 ASN 或城市等字段时为 `true`
- `queryFailed`：本轮查询失败但仍有旧缓存

## 6. 规则定义
### 6.1 地区名本地化
- 优先展示中文地区名
- 无映射时回退原始英文值
- 格式：`国家 · 城市` 或 `地区 · 城市`

### 6.2 IP 脱敏展示
- IPv4：默认展示前两段与最后一段，例如 `203.0.*.18`
- IPv6：默认展示前三段，例如 `2409:8a1c:7b0::*`
- Large 可通过开关显示完整 IP，默认仍脱敏

### 6.3 运营商字段截断
- Small：不显示运营商
- Medium：最多 16 个英文字符或 8 个中文字符
- Large：最多 24 个英文字符或 12 个中文字符

### 6.4 查询超时与缓存回退
- 默认超时 `2500ms`
- 查询失败时：
  - 有缓存：展示缓存并标记“上次身份”
  - 无缓存：进入错误态

## 7. 功能需求
### 7.1 核心功能
- 展示出口地区
- 展示脱敏 IP
- 展示 ASN/运营商摘要
- 展示 DNS 摘要与当前网络环境
- 判断是否匹配期望地区

### 7.2 增强功能
- Large 尺寸显示 DNS 列表摘要
- 支持显示当前网络为 Wi‑Fi 或蜂窝
- 支持点击跳转到 Egern 指定页面、IP 检测页或帮助页

### 7.3 不纳入 v1
- 地图底图
- 多出口对比
- 小组件内直接切换目标地区

## 8. 信息架构与多尺寸布局
### 8.1 统一信息层级
- 主焦点：出口地区
- 辅助指标：IP、运营商、DNS
- 状态提示：是否匹配期望地区、更新时间

### 8.2 `systemSmall`
- 顶部：标题或国家短码
- 中部：大号地区名，如 `东京`
- 底部：`JP 出口 · 匹配预期` 或 `身份已缓存`

### 8.3 `systemMedium`
- 左侧：地区、国家、匹配状态
- 右侧：IP、ASN/运营商两行信息
- 底部：DNS 摘要 + 更新时间

### 8.4 `systemLarge`
- 顶部：标题、策略名、状态徽标
- 中部：地点卡主内容，突出地区与城市
- 底部：
  - IP
  - ASN / 运营商
  - DNS 摘要
  - 当前网络环境（Wi‑Fi / 蜂窝）

### 8.5 `accessoryCircular`
- 中心：国家短码或城市短名
- 外圈颜色：匹配为冷蓝绿，不匹配为柔黄或柔红

### 8.6 `accessoryRectangular`
- 第一行：`日本 · 东京`
- 第二行：`203.0.*.18 · 匹配`

### 8.7 `accessoryInline`
- 单行：`出口身份：日本东京，匹配预期`

## 9. 视觉规范
### 9.1 视觉方向
- 更像 Apple Weather 的地点卡而不是技术仪表盘
- 以地点感和空间感为第一印象

### 9.2 主色板
- 主色：`#77A8FF`
- 匹配状态：`#5CC48D`
- 待确认：`#F4B65F`
- 异常：`#FF7D6E`
- 背景：冷灰蓝到海雾蓝渐变

### 9.3 视觉细节
- 地区名可作为唯一大字号焦点
- IP、ASN、DNS 采用轻量信息层
- 不使用大面积旗帜图形或地图装饰

### 9.4 留白与排版
- 国家/城市为标题层
- 技术字段必须短句化
- 每个尺寸最多保留一个状态色焦点

## 10. 环境变量（Env）
- `TITLE`：默认 `出口身份`
- `REFRESH_MINUTES`：默认 `30`
- `OPEN_URL`：点击跳转地址
- `ACCENT_MODE`：`auto` / `cool` / `mist`
- `STORAGE_KEY`：默认 `egress-identity-summary`
- `POLICY_NAME`：出口查询使用的策略
- `IP_INFO_API`：主查询接口
- `BACKUP_IP_INFO_API`：备用查询接口
- `REQUEST_TIMEOUT_MS`：默认 `2500`
- `EXPECTED_REGION`：可选，用于匹配判断
- `SHOW_FULL_IP`：默认 `false`
- `DNS_DISPLAY_MODE`：`summary` / `first` / `hidden`

## 11. 状态与异常处理
### 11.1 API 超时
- 有缓存：展示缓存并标记“使用上次身份”
- 无缓存：展示错误卡 `暂时无法确认出口身份`

### 11.2 返回字段缺失
- 缺 ASN：隐藏 ASN 行，不把 `null` 展示到界面
- 缺城市：回退为国家或地区
- 缺国家：进入 `partialData` 状态

### 11.3 地区解析失败
- 使用原始字符串回退
- 文案从“匹配/不匹配”降为“待确认”

### 11.4 缓存过期但网络不可用
- 继续展示旧缓存
- 状态区标记：`身份可能已过期`
- 颜色弱化，避免误判为实时

## 12. 验收标准
### 12.1 尺寸验收
- Small 只看地区也成立
- Medium 的技术字段不压迫主地点信息
- Large 可以扩展信息，但地点仍是第一视觉焦点

### 12.2 数据验收
- 正常查询：地区、IP、ASN 正确展示
- 部分字段缺失：自动回退，不出现原始空值
- 超时：缓存与错误态行为明确
- 期望地区比对：匹配状态准确

### 12.3 审美验收
- 看上去像系统地点卡，不像 IP 查询网页
- 文字层次清楚，技术字段不会喧宾夺主

### 12.4 实现验收
- 查询逻辑与缓存逻辑在 companion 脚本完成
- widget 只消费整理后的身份摘要

## 13. 开发拆解建议
1. 实现出口身份 schedule 查询脚本与缓存
2. 实现地区名本地化、IP 脱敏、期望地区比对
3. 实现 view model：地点、技术字段、状态文案
4. 实现多尺寸布局与 DNS/网络环境降级策略
5. 覆盖接口超时、字段缺失、缓存回退三类边界
