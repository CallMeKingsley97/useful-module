# 需求文档：今日流量战报小组件

## 1. 背景与目标
系统层的流量统计通常缺少“代理视角”。代理用户真正关心的是：**今天代理消耗了多少、主要消耗在哪、什么时候最猛**。今日流量战报要把这些信息做成一张日历式、日报式的小组件，而不是工程化字节面板。

**核心目标：**
- 用一个主数字回答“今天总共走了多少流量”
- 用上行/下行、峰值时段、Top 消耗项解释这个数字
- 当无法拿到稳定 App 维度时，明确降级为规则组或域名类别视角
- 以 Apple 风格日报卡呈现，不做表格式监控面板

## 2. 用户画像与使用场景
### 2.1 用户画像
- 代理重度用户：想知道自己今天的代理成本
- 流媒体用户：关心大流量消耗集中在哪
- 审美导向用户：希望小组件像“今日摘要”而不是技术统计页

### 2.2 使用场景
- 晚上看一眼主屏，知道今天代理总体消耗
- 网络套餐紧张时，快速确认是否出现异常大流量
- 某天 AI 或视频使用量暴增后，回看峰值时段与主要规则组

## 3. Egern 能力边界与数据来源
### 3.1 能力边界
- Egern 公开 JavaScript API 未明确提供“全局真实流量计数”的直接读取接口
- 因此 v1 不能假设 generic 小组件能直接读取系统级流量统计
- 本组件必须依赖 companion 统计脚本持续写入 `ctx.storage`

### 3.2 v1 数据模式
1. **观察模式（推荐）**
   - companion `request/response` 脚本记录受管流量的估算字节数
   - 下载优先使用 `Content-Length`
   - 上传优先使用请求体长度或已知请求大小
   - 统计规则组、域名类别、时间桶

2. **估算模式（降级）**
   - 若字节数不稳定，则以请求次数和响应头可得字节估算
   - 文案需标记为“估算战报”

3. **请求量模式（最低降级）**
   - 若无法可靠统计字节，降级为“今日请求量战报”
   - 主数字由总请求数替代总流量

### 3.3 App 维度说明
- 若当前数据链路无法稳定拿到 App 维度，v1 必须降级为：
  - `topRuleGroup`
  - `topCategory`
  - 或 `topDomainFamily`
- PRD 中不把 `topApp` 写成刚性必达项

## 4. 产品定义
### 4.1 一句话卖点
把代理流量做成一张今天就能看懂的日报卡。

### 4.2 核心输出
- 今日总流量
- 上行 / 下行
- 峰值时段
- 最耗流量的规则组或类别
- 统计模式与数据覆盖状态

## 5. 数据契约
### 5.1 推荐 storage key
- `daily-traffic-report`

### 5.2 推荐 storage 结构
```json
{
  "date": "2026-03-18",
  "timezone": "Asia/Shanghai",
  "mode": "observed",
  "updatedAt": "2026-03-18T22:10:00+08:00",
  "traffic": {
    "uploadBytes": 314572800,
    "downloadBytes": 2147483648,
    "totalBytes": 2462056448
  },
  "top": {
    "ruleGroup": "YouTube",
    "category": "视频流媒体",
    "app": null
  },
  "peakHour": {
    "hour": 21,
    "bytes": 734003200
  },
  "hourly": [
    { "hour": 9, "bytes": 52428800 },
    { "hour": 12, "bytes": 188743680 },
    { "hour": 21, "bytes": 734003200 }
  ],
  "requestCount": 428,
  "sampleCoverage": 0.82,
  "flags": {
    "stale": false,
    "zeroTraffic": false,
    "lowCoverage": false,
    "requestOnly": false
  }
}
```

### 5.3 字段规则
- `date`：按本地自然日切分
- `sampleCoverage`：`0 ~ 1`，表示受观测流量覆盖率估算
- `requestOnly=true`：代表已降级为请求量模式
- `hourly`：用于 Large 的简化日内热度分布

## 6. 统计规则
### 6.1 字节单位换算
- `< 1024`：B
- `< 1024 * 1024`：KB
- `< 1024 * 1024 * 1024`：MB
- 其余：GB
- 保留规则：
  - `< 10` 单位保留 1 位小数
  - `>= 10` 单位取整数

### 6.2 日期切日逻辑
- 以本地时区自然日为准
- 新的一天第一次写入时必须重置累计结构
- 若检测到旧 `date`，companion 脚本先归档再重建当日对象

### 6.3 峰值时段算法
- 以 1 小时为桶统计 `bytes`
- 取最大值所在小时为 `peakHour`
- 若并列，取**更早**的小时，保证结果稳定

### 6.4 Top 项命名压缩
- 中文超过 8 个字截断
- 英文超过 18 个字符截断
- Small 只显示一项 Top；Large 可额外显示类别

## 7. 功能需求
### 7.1 核心功能
- 展示今日总流量
- 展示上行/下行分布
- 展示峰值时段
- 展示最耗流量规则组或类别
- 展示统计模式与覆盖率状态

### 7.2 增强功能
- Large 展示 24 小时压缩热度
- 支持“估算战报”与“请求量战报”文案自动切换
- 支持点击跳转到 Egern 指定统计页或外部说明页

### 7.3 不纳入 v1
- 每应用排行详情
- 历史多日对比
- 套餐余量预测

## 8. 信息架构与多尺寸布局
### 8.1 统一信息层级
- 主焦点：今日总流量
- 辅助指标：上行/下行、峰值时段
- 状态提示：Top 项、统计模式、覆盖率

### 8.2 `systemSmall`
- 顶部：`今日流量`
- 中部：大号总量，如 `2.3 GB`
- 底部：`峰值 21:00 · YouTube`

### 8.3 `systemMedium`
- 左侧：总流量与一句摘要
- 右侧：上行 / 下行双层结构
- 底部：峰值时段 + Top 规则组

### 8.4 `systemLarge`
- 顶部：标题、日期、模式标记
- 中部：主流量数字 + 上下行拆分
- 底部：
  - 峰值时段
  - Top 规则组 / 类别
  - 日内热度条
  - 覆盖率说明

### 8.5 `accessoryCircular`
- v1 不提供
- 原因：总量、峰值、Top 三者无法在圆形规格中同时成立

### 8.6 `accessoryRectangular`
- 第一行：`今日 2.3 GB`
- 第二行：`峰值 21:00 · YouTube`

### 8.7 `accessoryInline`
- 单行：`今日流量：2.3 GB，峰值 21:00`

## 9. 视觉规范
### 9.1 视觉方向
- 气质接近 Apple 的“今日摘要”卡片
- 强调节奏感和日报感，而不是财务报表感

### 9.2 主色板
- 主色：`#77A8FF`
- 下载：`#5B8CFF`
- 上传：`#8ED1A5`
- 提示：`#F4B65F`
- 异常：`#FF7D6E`
- 背景：低饱和晨雾蓝到深海军蓝渐变

### 9.3 图形语言
- 主数字极简
- 上下行用细条形或双色胶囊条表示
- 日内热度仅做弱存在感，不做复杂图表

### 9.4 排版与留白
- 主数字必须足够呼吸
- 次级信息最多两层
- Top 项文案优先短句，不用技术列表

## 10. 环境变量（Env）
- `TITLE`：默认 `今日流量`
- `REFRESH_MINUTES`：默认 `15`
- `OPEN_URL`：点击跳转地址
- `ACCENT_MODE`：`auto` / `cool` / `daylight`
- `STORAGE_KEY`：默认 `daily-traffic-report`
- `TIMEZONE`：默认本地时区
- `BYTE_MODE`：`observed` / `estimated` / `auto`
- `SHOW_COVERAGE`：默认 `true`
- `MIN_COVERAGE`：默认 `0.6`
- `PEAK_BUCKET_MINUTES`：默认 `60`
- `TOP_LABEL_MODE`：`ruleGroup` / `category` / `auto`

## 11. 状态与异常处理
### 11.1 当日无数据
- 主文案：`今天还没有流量`
- 次文案：`等待统计脚本写入`

### 11.2 零流量
- 仍视为正常状态，不标红
- 采用安静的中性色表现

### 11.3 切日后首次刷新
- 若 `date` 已切换但暂无新数据，展示 `新的一天开始了`
- 不继续显示前一天累计值

### 11.4 历史残留脏数据
- companion 脚本需按 `date` 严格隔离
- widget 发现 `date` 不匹配时直接进入空状态

### 11.5 低覆盖率
- 当 `sampleCoverage < MIN_COVERAGE` 时：
  - 文案改为 `估算战报`
  - Large 底部补充 `覆盖率有限`

## 12. 验收标准
### 12.1 尺寸验收
- Small 只看总量也成立
- Medium 能解释总量从哪来
- Large 能补足峰值和 Top 项，不变成报表墙

### 12.2 数据验收
- 正常数据：总量、上下行、Top、峰值正确
- 空数据：显示等待态
- 低覆盖率：正确切换为估算文案
- `requestOnly`：正确改为请求量模式

### 12.3 审美验收
- 截图在 iOS 主屏上像日报卡，而不是网络监控表
- 颜色克制，主数字始终是视觉中心

### 12.4 实现验收
- companion 脚本负责统计与切日逻辑
- widget 只读取当日摘要并完成格式化渲染

## 13. 开发拆解建议
1. 实现按自然日累积的 traffic storage 契约
2. 实现字节统计、Top 项统计、峰值桶统计
3. 实现格式化层：单位换算、摘要文案、模式切换
4. 实现 Small / Medium / Large / accessory 布局
5. 验证空数据、零流量、切日、低覆盖率、请求量模式
